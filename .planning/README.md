# `.planning/` — 任务规划与设计文档

本目录存放 paperterm 所有**规划性**文档（任务规划、设计方案、阶段拆解），**不放代码**。

## 命名规范

```
.planning/{YYYYMMDD}_{task_name}.md
```

- `YYYYMMDD`：起草日期（八位数字）
- `task_name`：任务的简短英文/拼音/中文别名，避免和同日其它任务重名
- 例：`.planning/20260508_init_vibe_coding.md`、`.planning/20260508_paperterm_v0.1_design.md`

## 文件格式

参见 `.agent-rules/approval.md`（先规划后执行规则）。**每个任务规划文件**应包含：

- `## 任务` 标题
- `**背景**` / `**影响范围**` / `**前置条件**`
- 若干 `### Stage N` 段（含 **目标** / **成功标准** / **状态**）
- `## 待确认事项`（执行前需用户决策的开放问题）

> 例外：设计文档（如 `20260508_paperterm_v0.1_design.md`）描述的是产品/技术方案而非任务拆解，不必套用上述模板，按设计文档自身需要组织。

## 生命周期

- **追加，不删**。规划文件视为只读历史，即使方案被推翻也保留原文，新增的修订写在文件末尾的 `## 修订记录` 节
- 每个 Stage 完成后即时更新该 Stage 的 `**状态**`：`Not Started` → `In Progress` → `Complete`
- 整个任务结束后，在文件末尾追加 `## 完成记录` 节，记录完成时间、与原计划的偏差、最终 commit hash
- 历史规划文档**永久保留**，作为决策审计与新成员上手的入口

## 与会话状态的边界

`.cursor/task_plan.md` / `progress.md` / `findings.md` 是**会话级临时**状态，记录当下正在跑的执行细节，不入 git；`.cursor/memory/`（含 `MEMORY.md` 索引）是**本机持久记忆**，也不入 git。**本目录是任务级永久记录，入 git**，作为决策审计与跨会话恢复的入口。
